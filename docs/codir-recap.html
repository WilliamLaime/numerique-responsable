1L'extension en 30 secondes

Extension Chrome Manifest V3 avec Side Panel qui audite un site web sur deux axes complémentaires : la conformité RGAA 4.1 WCAG 2.1 A/AA et l'écoconception RGESN 2024.

106
critères RGAA sur 13 thématiques
91
checks automatiques
24
règles RGESN / 9 thématiques
1 → N
pages auditables (crawl sitemap)

L'extension découvre les URLs du site via sitemap.xml + liens internes, puis orchestre un pool de workers parallèles (1 à 8 onglets simultanés, limite 5/10/25/50 ou illimitée). Les résultats sont filtrables par statut et thématique, triables selon trois vues (thématique / règle / page), exportables en CSV et PDF, et sauvegardables localement avec un système de jump-to-element qui surligne l'élément fautif directement sur la page auditée.

2.3 Scoring

Score Tanaguru-like : C / (C + NC) × 100, NA et NT exclus. Seuils d'affichage : ≥ 75 vert, 50–74 orange, < 50 rouge. Agrégation multi-pages : pire statut par règle (NC > NT > C > NA) — cohérent avec l'approche d'un référentiel officiel.

Ce qu'il faut savoir pour le CoDir
  • 100 % local — aucune donnée ne quitte le navigateur de l'utilisateur.
  • Pas de cloud, pas d'API tierce, pas de télémétrie.
  • Un utilisateur = un silo — pas de partage entre navigateurs / machines.
  • Export CSV / PDF (génération côté client) = seule voie pour partager un audit.
Point d'attention RGPD
Le DOM capturé dans les samples peut contenir des données sensibles si on audite des pages internes authentifiées. À clarifier dans la politique d'usage interne.

4Points forts à défendre

Différenciants majeurs
  1. Seul outil du marché à combiner RGAA + RGESN dans une même interface. Les concurrents traitent l'un OU l'autre.
  2. Référentiel français natif — pas une traduction approximative de WCAG, mais la terminologie RGAA 4.1 directement exploitable pour la déclaration d'accessibilité légale.
  3. Crawl multi-pages intégré — Lighthouse et axe DevTools fonctionnent 1 page à la fois, nous scannons N pages en parallèle avec découverte automatique.
  4. Calcul de contraste WCAG 2.0 propre — luminance sRGB, alpha-blending, remontée de la chaîne des fonds. Au niveau d'axe-core.
  5. Algorithme WAI-ARIA 1.2 complet pour le nom accessible (aria-labelledby, aria-label, <label for>, alt, title, contenu récursif).
  6. Confidentialité by design — 100 % local, argument fort en contexte bancaire / santé / secteur public.
  7. UX moderne — Side Panel MV3, jump-to-element, export PDF/CSV, historique des audits.
  8. Manifest V3 — pérennité assurée (MV2 sera désactivé par Chrome).

5Points à assumer en transparence

Qualité du code

Dette technique identifiée
  • Monolithes — audit.js (1 430 lignes) et sidepanel.js (1 450 lignes) non modularisés → testabilité unitaire limitée.
  • Pas de CI/CD — aucun workflow GitHub Actions, aucune validation automatique.
  • Commits peu explicites (« suite », « à continuer ») — traçabilité limitée.
  • Aucun tag de version, pas de CHANGELOG.

Couverture fonctionnelle

Limites intrinsèques à assumer
  • Pas de tests clavier ni interaction réelle — limite commune à tous les outils statiques (axe, Lighthouse, Wave incluses).
  • SPA partiellement couvertes — état au chargement capturé, pas post-interaction (modales, changements de route).
  • Heuristiques regex sur quelques critères (alt générique, transcription, captcha) → faux positifs possibles.
  • Dark mode CSS (prefers-color-scheme) non couvert pour le contraste.
  • 52 critères RGAA manuels — assumé par le référentiel. Aucun outil automatique ne couvre 100 %.

Écoconception — honnêteté scientifique

Périmètre RGESN
  • Pas de calcul d'empreinte carbone (contrairement à Website Carbon, Ecoindex). On mesure des indicateurs techniques, pas des grammes de CO₂.
  • Seuils fixes — les valeurs (2 Mo, 1500 nœuds) viennent des bonnes pratiques RGESN mais ne tiennent pas compte du contexte applicatif.
  • Visite 1 vs visite N — poids mesuré sans cache, donc on mesure le pire cas et non la moyenne utilisateur.

Roadmap naturelle à évoquer

  1. Modularisation du moteur en plugins de règles testables unitairement.
  2. CI GitHub Actions + couverture de code Istanbul.
  3. Rapports comparatifs entre audits (delta avant / après).
  4. Partage d'audits via lien ou export JSON (aujourd'hui silo par navigateur).
  5. Intégration optionnelle d'un calcul CO₂ (Sustainable Web Design formula).
  6. Mode audit authentifié / connecté pour pages internes.

7Positionnement stratégique

Messages clés pour le CoDir
  1. Outil sans équivalent direct — extension légère qui combine RGAA + RGESN sur le marché français.
  2. Complémentaire aux outils existants, pas substitut. On facilite l'entrée de gamme : sensibilisation, pré-audit, suivi dans le temps.
  3. Confidentialité native 100 % local — argument fort en contexte bancaire, santé, secteur public.
  4. Alignement obligations légales RGAA REEN — loi du 11 février 2005 / article 47 pour le secteur public et grandes entreprises ; loi REEN 2021 en montée en charge.
  5. Clarification attendue du CoDir — positionnement (outil interne ? produit diffusable ? open source ?). Cela détermine la roadmap : CI/CD, modularisation, support utilisateurs.

Préparation de la présentation — 5 actions concrètes

  1. Lancer l'extension sur le site corporate, exporter le PDF → visuel parlant.
  2. Lancer Lighthouse sur le même site → montrer la complémentarité, pas le chevauchement.
  3. Préparer 2–3 slides chiffrées : « X critères RGAA couverts », « Y Mo détectés », « Z % conformité ».
  4. Anticiper la question RGPD → réponse stockage local + roadmap export/partage.
  5. Anticiper la question roadmap/budget → 3 chantiers : modularisation + CI/CD + calcul CO₂.
